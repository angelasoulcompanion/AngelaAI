## 1. บทนำ

### 1.1 วัตถุประสงค์
เอกสารฉบับนี้กำหนดนโยบายระบบบริหารจัดการความปลอดภัยของข้อมูล (ISMS) สำหรับบริษัท Orbplus จำกัด นโยบายเหล่านี้เป็นกรอบสำหรับการปกป้องทรัพย์สินข้อมูลและรับประกันการรักษาความลับ ความสมบูรณ์ และความพร้อมใช้งานของข้อมูล

### 1.2 การบังคับใช้
คู่มือนโยบายนี้ใช้กับ:
- พนักงานทั้งหมดของบริษัท Orbplus จำกัด
- ผู้รับเหมา ที่ปรึกษา และพนักงานชั่วคราวทั้งหมด
- ผู้ให้บริการบุคคลที่สามทั้งหมดที่เข้าถึงข้อมูลของ Orbplus
- ทรัพย์สินข้อมูลทั้งหมดที่ Orbplus เป็นเจ้าของ ดำเนินการ หรือจัดการ

### 1.3 การปฏิบัติตาม
การปฏิบัติตามนโยบายเหล่านี้เป็นข้อบังคับ การละเมิดอาจส่งผลให้เกิดการลงโทษทางวินัย จนถึงและรวมถึงการยกเลิกการจ้างงานหรือสัญญา และอาจต้องรับผิดทางกฎหมาย

### 1.4 เอกสารที่เกี่ยวข้อง

| เอกสาร | คำอธิบาย |
|--------|---------|
| ORB-SOP-Manual | คู่มือขั้นตอนการปฏิบัติงานมาตรฐาน |
| ORB-RA-2025-001 | รายงานการประเมินความเสี่ยง |
| ORB-SOA-001 | คำชี้แจงการบังคับใช้ |
| ORB-BCP-001 | แผนความต่อเนื่องทางธุรกิจ |

## 2. นโยบายความปลอดภัยของข้อมูล

### 2.1 คำประกาศนโยบาย

**บริษัท Orbplus จำกัด มุ่งมั่นที่จะปกป้องทรัพย์สินข้อมูลของตนและของลูกค้าจากภัยคุกคามทั้งหมด ไม่ว่าจะเป็นภายในหรือภายนอก โดยเจตนาหรือโดยอุบัติเหตุ**

ความปลอดภัยของข้อมูลมีความจำเป็นในการรักษา:
- ความมั่นใจและความไว้วางใจของลูกค้า
- การดำเนินธุรกิจและความต่อเนื่อง
- ความได้เปรียบในการแข่งขัน
- การปฏิบัติตามกฎหมายและข้อบังคับ
- ชื่อเสียงองค์กร

### 2.2 วัตถุประสงค์ด้านความปลอดภัย

| วัตถุประสงค์ | คำอธิบาย | มาตรการ |
|------------|---------|--------|
| **การรักษาความลับ** | ข้อมูลสามารถเข้าถึงได้เฉพาะบุคคลที่ได้รับอนุญาต | การควบคุมการเข้าถึง, การเข้ารหัส |
| **ความสมบูรณ์** | ข้อมูลถูกต้องและครบถ้วน | การตรวจสอบข้อมูล, บันทึกการตรวจสอบ |
| **ความพร้อมใช้งาน** | ข้อมูลพร้อมใช้งานเมื่อจำเป็น | ความซ้ำซ้อน, การสำรองข้อมูล, DR |

### 2.3 หลักการความปลอดภัย

1. **การป้องกันหลายชั้น** - มีมาตรการควบคุมความปลอดภัยหลายชั้น
2. **สิทธิ์ขั้นต่ำ** - เข้าถึงได้เท่าที่จำเป็นสำหรับหน้าที่งาน
3. **การแบ่งแยกหน้าที่** - หน้าที่สำคัญแบ่งระหว่างบุคลากร
4. **ความจำเป็นต้องรู้** - แบ่งปันข้อมูลเฉพาะเมื่อจำเป็น
5. **ความปลอดภัยโดยการออกแบบ** - ความปลอดภัยถูกสร้างในระบบตั้งแต่เริ่มต้น

### 2.4 ความมุ่งมั่นของฝ่ายบริหาร

ฝ่ายบริหารมุ่งมั่นที่จะ:
- จัดหาทรัพยากรที่เพียงพอสำหรับความปลอดภัยของข้อมูล
- จัดตั้งและรักษา ISMS
- สร้างความตระหนักด้านความปลอดภัยทั่วทั้งองค์กร
- ทบทวนประสิทธิภาพ ISMS อย่างสม่ำเสมอ
- ปรับปรุงท่าทีด้านความปลอดภัยอย่างต่อเนื่อง

## 3. ขอบเขต ISMS

### 3.1 ขอบเขตองค์กร

| องค์ประกอบ | รายละเอียด |
|----------|----------|
| **องค์กร** | บริษัท Orbplus จำกัด |
| **กิจกรรมทางธุรกิจ** | การพัฒนา ปรับใช้ และสนับสนุนโซลูชัน ERP บนคลาวด์ |
| **ที่ตั้ง** | สำนักงานใหญ่และสภาพแวดล้อมการทำงานระยะไกล |
| **ผลิตภัณฑ์/บริการ** | Orbplus ERP, การเชื่อมต่อ API, การสนับสนุนลูกค้า |

### 3.2 ทรัพย์สินข้อมูลในขอบเขต

| หมวดหมู่ทรัพย์สิน | ตัวอย่าง |
|----------------|--------|
| **ข้อมูล** | ข้อมูลลูกค้า, ซอร์สโค้ด, บันทึกธุรกิจ |
| **ซอฟต์แวร์** | แอปพลิเคชัน ERP, เครื่องมือพัฒนา, ระบบปฏิบัติการ |
| **ฮาร์ดแวร์** | โครงสร้างพื้นฐานคลาวด์, เวิร์กสเตชัน, อุปกรณ์เครือข่าย |
| **บริการ** | โฮสติ้งคลาวด์, อีเมล, แพลตฟอร์มพัฒนา |
| **บุคลากร** | พนักงาน, ผู้รับเหมา, พันธมิตรเอาท์ซอร์ส |

### 3.3 การยกเว้น

| การยกเว้น | เหตุผล |
|---------|-------|
| โครงสร้างพื้นฐาน IT ของลูกค้า | อยู่นอกเหนือการควบคุมของ Orbplus |
| อุปกรณ์ส่วนตัวของผู้ใช้ | ไม่ได้จัดการโดยบริษัท |

### 3.4 การเชื่อมต่อและการพึ่งพา

| การเชื่อมต่อ | คำอธิบาย | การควบคุม |
|------------|---------|----------|
| ผู้ให้บริการคลาวด์ | โฮสติ้ง AWS/GCP | ข้อกำหนดความปลอดภัยในสัญญา |
| พันธมิตรเอาท์ซอร์ส | TERA, INTENSE, INET | ข้อตกลงด้านความปลอดภัย, การควบคุมการเข้าถึง |
| ลูกค้า | ผู้ใช้ SaaS | เงื่อนไขการใช้บริการ, การปกป้องข้อมูล |

## 4. ภาวะผู้นำและความมุ่งมั่น

### 4.1 ความรับผิดชอบของผู้บริหารระดับสูง

CEO และทีมบริหารจะต้อง:

1. กำหนดนโยบายและวัตถุประสงค์ด้านความปลอดภัยของข้อมูล
2. ให้มั่นใจว่าข้อกำหนด ISMS ถูกรวมเข้ากับกระบวนการทางธุรกิจ
3. ให้มั่นใจว่ามีทรัพยากรเพียงพอ
4. สื่อสารความสำคัญของการจัดการความปลอดภัยที่มีประสิทธิภาพ
5. ให้มั่นใจว่า ISMS บรรลุผลลัพธ์ที่ตั้งใจไว้
6. กำกับและสนับสนุนการปรับปรุงอย่างต่อเนื่อง
7. สนับสนุนบทบาทการจัดการที่เกี่ยวข้องอื่นๆ

### 4.3 คณะกรรมการความปลอดภัย

**สมาชิก:**
- CEO (ประธาน)
- หัวหน้าฝ่ายผลิตภัณฑ์และเทคโนโลยี
- หัวหน้าฝ่ายการเงินและธุรการ
- เจ้าหน้าที่ความปลอดภัย (เลขานุการ)

**ความรับผิดชอบ:**
- ทบทวนและอนุมัตินโยบายความปลอดภัย
- กำกับดูแลกิจกรรมการจัดการความเสี่ยง
- ทบทวนเหตุการณ์และแนวโน้มด้านความปลอดภัย
- อนุมัติการลงทุนด้านความปลอดภัย
- เฝ้าระวังประสิทธิภาพ ISMS

**ความถี่ในการประชุม:** รายไตรมาส (หรือตามความจำเป็น)

| กิจกรรม | CEO | เจ้าหน้าที่ความปลอดภัย | หัวหน้าเทคโนโลยี | หัวหน้าแผนก | พนักงาน |

## 6. นโยบายการจัดการความเสี่ยง

4. **การประเมินความเสี่ยง** - เปรียบเทียบกับเกณฑ์ความเสี่ยง
5. **การจัดการความเสี่ยง** - เลือกและดำเนินการควบคุม
6. **เฝ้าระวังและทบทวน** - การเฝ้าระวังและการประเมินใหม่อย่างต่อเนื่อง

## 7. นโยบายการควบคุมการเข้าถึง

### 7.1 คำประกาศนโยบาย

### 7.2 หลักการควบคุมการเข้าถึง

1. **ปฏิเสธโดยค่าเริ่มต้น** - ปฏิเสธการเข้าถึงเว้นแต่ได้รับอนุญาตอย่างชัดเจน

4. **การแบ่งแยกหน้าที่** - แยกหน้าที่ที่ขัดแย้งกัน

- ตัวระบุเฉพาะ

- เจ้าของและผู้ดูแล
- ที่ตั้ง

| **ลับ** | ผลกระทบต่อธุรกิจอย่างมีนัยสำคัญ | ควบคุมการเข้าถึง, จัดเก็บปลอดภัย |

| ลับ | เครื่องหมาย "ลับ" | ที่ตั้งปลอดภัย | ช่องทางปลอดภัย | ทำลาย/ล้าง |
| ภายใน | "ภายใน" ตัวเลือก | มาตรฐาน | มาตรฐาน | ทำลายปกติ |
| สาธารณะ | ไม่จำเป็น | ใดก็ได้ | ใดก็ได้ | ทำลายปกติ |

### 8.5 การใช้งานที่ยอมรับได้

#### 8.5.1 การใช้งานที่อนุญาต
- วัตถุประสงค์ทางธุรกิจที่สอดคล้องกับหน้าที่งาน

- การใช้งานที่ปฏิบัติตามใบอนุญาตและข้อตกลง

#### 8.5.2 การใช้งานที่ห้าม
- การเข้าถึงหรือดึงข้อมูลโดยไม่ได้รับอนุญาต
- การติดตั้งซอฟต์แวร์ที่ไม่ได้รับอนุญาต
- การแชร์ข้อมูลรับรองหรือหลีกเลี่ยงการควบคุม
- กิจกรรมที่ผิดกฎหมายหรือการคุกคาม
- การจัดเก็บข้อมูลส่วนตัวในระบบของบริษัท

### 8.6 การคืนทรัพย์สิน
ทรัพย์สินของบริษัททั้งหมดต้องคืนเมื่อ:
- สิ้นสุดการจ้างงาน
- สิ้นสุดสัญญา
- เปลี่ยนบทบาท (ถ้ามี)
- ตามคำขอของฝ่ายบริหาร

## 10. นโยบายความปลอดภัยทางกายภาพและสิ่งแวดล้อม

### 10.1 คำประกาศนโยบาย
การเข้าถึงทางกายภาพต่อสิ่งอำนวยความสะดวกและอุปกรณ์จะถูกควบคุมเพื่อป้องกันการเข้าถึงโดยไม่ได้รับอนุญาต ความเสียหาย หรือการรบกวน

### 10.2 พื้นที่ปลอดภัย

#### 10.2.1 ขอบเขตความปลอดภัย
| โซน | คำอธิบาย | การควบคุม |
|-----|---------|----------|
| สาธารณะ | ต้อนรับ, ล็อบบี้ | ลงทะเบียนผู้เยี่ยมชม |
| สำนักงาน | พื้นที่ทำงานทั่วไป | บัตรเข้าออก |
| จำกัด | ห้องเซิร์ฟเวอร์, คลังเอกสาร | บัตรเข้าออก + PIN |

#### 10.2.2 การควบคุมการเข้า
| การควบคุม | ข้อกำหนด |
|---------|---------|
| การเข้าถึงทางกายภาพ | ต้องมีบัตร/ป้าย |
| การจัดการผู้เยี่ยมชม | ลงทะเบียน, ต้องมีผู้พา |
| พื้นที่รับสินค้า | ควบคุม, เฝ้าระวัง |
| ทางออกฉุกเฉิน | มีสัญญาณเตือน, สำหรับฉุกเฉินเท่านั้น |

### 10.3 ความปลอดภัยของอุปกรณ์

#### 10.3.1 การวางอุปกรณ์
- เซิร์ฟเวอร์ในที่ตั้งที่ปลอดภัย ควบคุมอุณหภูมิ
- จอภาพวางเพื่อป้องกันการแอบดู
- อุปกรณ์ละเอียดอ่อนในพื้นที่จำกัด

#### 10.3.2 สาธารณูปโภคสนับสนุน
| สาธารณูปโภค | การป้องกัน |
|-----------|----------|
| ไฟฟ้า | UPS, ป้องกันไฟกระชาก |
| ความเย็น | ควบคุมอุณหภูมิ, เฝ้าระวัง |
| สายเคเบิล | ป้องกัน, ติดฉลาก |

#### 10.3.3 การบำรุงรักษา
- เฉพาะบุคลากรที่ได้รับอนุญาต
- รักษาบันทึกการบำรุงรักษา
- ล้างข้อมูลก่อนการทำลาย

### 10.4 โต๊ะและหน้าจอว่าง

| ข้อกำหนด | การดำเนินการ |
|---------|-------------|
| โต๊ะว่าง | ล็อคเอกสารเมื่อออกจากที่ |
| หน้าจอว่าง | ล็อคอัตโนมัติหลัง 5 นาที |
| เครื่องพิมพ์ | เก็บเอกสารที่พิมพ์ทันที |
| กระดานไวท์บอร์ด | ลบข้อมูลละเอียดอ่อน |

## 11. นโยบายความปลอดภัยด้านปฏิบัติการ

### 11.1 คำประกาศนโยบาย
การดำเนินงานประจำวันจะต้องดำเนินการอย่างปลอดภัย พร้อมการควบคุมที่เหมาะสมเพื่อป้องกัน Malware การสูญเสียข้อมูล และความล้มเหลวในการดำเนินงาน

### 11.2 ขั้นตอนการดำเนินงาน

#### 11.2.1 ขั้นตอนที่บันทึก
การดำเนินงานที่สำคัญทั้งหมดจะต้องบันทึก รวมถึง:
- งานการดูแลระบบ
- ขั้นตอนการสำรองและกู้คืน
- ขั้นตอนการตอบสนองต่อเหตุการณ์
- ขั้นตอนการจัดการการเปลี่ยนแปลง

#### 11.2.2 การจัดการการเปลี่ยนแปลง
| ประเภทการเปลี่ยนแปลง | ระยะเวลาล่วงหน้า | การอนุมัติ |
|-------------------|---------------|---------|
| มาตรฐาน | 24 ชั่วโมง | อนุมัติอัตโนมัติ |
| ปกติ | 5 วัน | CAB |
| ฉุกเฉิน | ทันที | CAB ฉุกเฉิน |

### 11.3 การป้องกัน Malware

| การควบคุม | การดำเนินการ |
|---------|-------------|
| การป้องกัน Endpoint | EDR บน Endpoint ทั้งหมด |
| การกรองอีเมล | Anti-malware, Anti-phishing |
| การกรองเว็บ | บล็อก URL ที่เป็นอันตราย |
| การอัปเดต | อัปเดต Signature เป็นประจำ |
| ความตระหนัก | อบรมผู้ใช้เกี่ยวกับภัยคุกคาม |

### 11.4 การสำรองข้อมูล

#### 11.4.1 ข้อกำหนดการสำรองข้อมูล

| ประเภทข้อมูล | ความถี่ | การเก็บรักษา | ที่ตั้ง |
|------------|--------|------------|-------|
| ฐานข้อมูล Production | 6 ชั่วโมง | 30 วัน | หลัก + DR |
| บันทึกธุรกรรม | 15 นาที | 7 วัน | หลัก + DR |
| โค้ดแอปพลิเคชัน | ทุก Commit | ไม่มีกำหนด | Git repository |
| การตั้งค่า | รายวัน | 90 วัน | สำรองปลอดภัย |
| อิมเมจระบบ | รายสัปดาห์ | 4 สัปดาห์ | ไซต์ DR |

#### 11.4.2 การทดสอบการสำรองข้อมูล
| การทดสอบ | ความถี่ |
|--------|--------|
| การกู้คืนตัวอย่าง | รายสัปดาห์ |
| การทดสอบกู้คืนเต็มรูปแบบ | รายเดือน |
| การทดสอบ DR failover | รายไตรมาส |

### 11.5 การบันทึกและเฝ้าระวัง

#### 11.5.1 เหตุการณ์ที่ต้องบันทึก

| ประเภทเหตุการณ์ | ตัวอย่าง |
|---------------|--------|
| เหตุการณ์การเข้าถึง | ล็อกอิน, ล็อกเอาต์, ความพยายามที่ล้มเหลว |
| การดำเนินการสิทธิ์พิเศษ | กิจกรรมผู้ดูแลระบบ, การเปลี่ยนแปลงการตั้งค่า |
| เหตุการณ์ความปลอดภัย | การแจ้งเตือน, เหตุการณ์, การละเมิดนโยบาย |
| เหตุการณ์ระบบ | ข้อผิดพลาด, ความล้มเหลว, ประสิทธิภาพ |

#### 11.5.2 การป้องกันบันทึก
- บันทึกจัดเก็บในที่ตั้งที่ป้องกันการแก้ไข
- การเข้าถึงจำกัดเฉพาะบุคลากรที่ได้รับอนุญาต
- เก็บรักษาขั้นต่ำ 1 ปี
- ทบทวนและแจ้งเตือนเป็นประจำ

### 11.6 การจัดการช่องโหว่

| กิจกรรม | ความถี่ | ความรับผิดชอบ |
|--------|--------|--------------|
| การสแกนช่องโหว่ | รายเดือน | เจ้าหน้าที่ความปลอดภัย |
| การประเมินแพตช์ | รายสัปดาห์ | DevOps |
| แพตช์วิกฤต | ภายใน 72 ชั่วโมง | DevOps |
| Penetration Testing | รายปี | ผู้ขายภายนอก |

## 12. นโยบายความปลอดภัยด้านการสื่อสาร

### 12.1 คำประกาศนโยบาย
ข้อมูลที่ส่งผ่านเครือข่ายจะต้องได้รับการป้องกันจากการดักจับ การแก้ไข และการเข้าถึงโดยไม่ได้รับอนุญาต

### 12.2 ความปลอดภัยเครือข่าย

#### 12.2.1 การควบคุมเครือข่าย

| การควบคุม | การดำเนินการ |
|---------|-------------|
| การแบ่งส่วน | แยก Dev/Prod/Management |
| Firewall | ขอบเขตและภายใน |
| IDS/IPS | การเฝ้าระวังเครือข่าย |
| VPN | การเข้ารหัสการเข้าถึงระยะไกล |
| การป้องกัน DDoS | การบรรเทาบน Cloud |

#### 12.2.2 การเข้าถึงเครือข่าย

| ประเภทการเข้าถึง | ข้อกำหนด |
|----------------|---------|
| เครือข่ายภายใน | การยืนยันตัวตนพนักงาน |
| การเข้าถึงระยะไกล | VPN + MFA |
| การเข้าถึงแขก | เครือข่ายแยก |
| ทรัพยากร Cloud | VPN หรือ API ที่ปลอดภัย |

### 12.3 การส่งข้อมูล

#### 12.3.1 การส่งทางอิเล็กทรอนิกส์

| การจำแนกข้อมูล | วิธีการส่ง |
|--------------|----------|
| จำกัด | ช่องทางเข้ารหัสเท่านั้น |
| ลับ | ช่องทางปลอดภัย (TLS 1.2+) |
| ภายใน | โปรโตคอลปลอดภัยมาตรฐาน |
| สาธารณะ | ไม่มีข้อจำกัด |

#### 12.3.2 ความปลอดภัยอีเมล

| การควบคุม | การดำเนินการ |
|---------|-------------|
| การเข้ารหัส | TLS สำหรับการส่ง |
| ไฟล์แนบ | การสแกน, จำกัดขนาด |
| อีเมลภายนอก | แบนเนอร์เตือน |
| ข้อมูลละเอียดอ่อน | ต้องเข้ารหัส |

### 12.4 การสื่อสารการพัฒนาที่ปลอดภัย

| การสื่อสาร | ความปลอดภัย |
|----------|------------|
| คลังโค้ด | การเข้าถึงที่ยืนยันตัวตน, เข้ารหัส |
| CI/CD pipelines | ปลอดภัย, บันทึกการตรวจสอบ |
| การสื่อสาร API | TLS, การยืนยันตัวตน |
| การจัดการ Secrets | Vault/จัดเก็บเข้ารหัส |

## 13. นโยบายการจัดหาและพัฒนาระบบ

### 13.1 คำประกาศนโยบาย
ความปลอดภัยจะถูกรวมเข้าในวงจรการพัฒนาตั้งแต่ความต้องการจนถึงการปรับใช้และการบำรุงรักษา

### 13.2 ข้อกำหนดความปลอดภัย

#### 13.2.1 การวิเคราะห์ความต้องการ
ระบบ/ฟีเจอร์ใหม่ทั้งหมดจะรวม:
- ข้อกำหนดความต้องการความปลอดภัย
- ข้อกำหนดการจำแนกข้อมูล
- ข้อกำหนดการควบคุมการเข้าถึง
- ข้อกำหนดการปฏิบัติตาม

#### 13.2.2 ความปลอดภัยโดยการออกแบบ
- การสร้างแบบจำลองภัยคุกคามสำหรับฟีเจอร์ใหม่
- การทบทวนสถาปัตยกรรมความปลอดภัย
- การประเมินผลกระทบต่อความเป็นส่วนตัว (ถ้ามี)

### 13.3 การพัฒนาที่ปลอดภัย

#### 13.3.1 มาตรฐานการเขียนโค้ด

| ข้อกำหนด | การดำเนินการ |
|---------|-------------|
| การตรวจสอบ Input | ตรวจสอบ Input ทั้งหมด |
| การเข้ารหัส Output | ป้องกันการโจมตี Injection |
| การยืนยันตัวตน | แข็งแกร่ง, รวมศูนย์ |
| การจัดการ Session | Token ที่ปลอดภัย, Timeout |
| การจัดการข้อผิดพลาด | ไม่มีข้อมูลละเอียดอ่อนในข้อผิดพลาด |
| การเข้ารหัส | ใช้ไลบรารีที่อนุมัติ |

#### 13.3.2 การปฏิบัติตาม OWASP Top 10
แอปพลิเคชันทั้งหมดจะต้องได้รับการป้องกันจาก:
1. การควบคุมการเข้าถึงที่ไม่ดี
2. ความล้มเหลวด้านการเข้ารหัส
3. Injection
4. การออกแบบที่ไม่ปลอดภัย
5. การตั้งค่าความปลอดภัยผิด
6. Components ที่มีช่องโหว่
7. ความล้มเหลวในการยืนยันตัวตน
8. ความล้มเหลวด้านความสมบูรณ์
9. ความล้มเหลวในการบันทึก
10. Server-Side Request Forgery

### 13.4 การตรวจสอบโค้ดและการทดสอบ

| กิจกรรม | เมื่อไหร่ | ใคร |
|--------|--------|-----|
| การตรวจสอบโค้ดโดยเพื่อนร่วมงาน | ทุก PR | นักพัฒนา |
| การตรวจสอบความปลอดภัย | การเปลี่ยนแปลงที่เกี่ยวกับความปลอดภัย | Security Champion |
| การวิเคราะห์แบบ Static | ทุกการ Build | อัตโนมัติ |
| การทดสอบแบบ Dynamic | ก่อน Release | QA |
| Penetration Testing | รายปี/Release ใหญ่ | ภายนอก |

### 13.5 ความปลอดภัยในการปรับใช้

| การควบคุม | ข้อกำหนด |
|---------|---------|
| การแยกสภาพแวดล้อม | Dev/Staging/Production |
| การจัดการการตั้งค่า | Infrastructure as Code |
| การจัดการ Secrets | ไม่มี Secrets ในโค้ด |
| การอนุมัติการปรับใช้ | กระบวนการจัดการการเปลี่ยนแปลง |
| ความสามารถในการย้อนกลับ | ขั้นตอนการย้อนกลับที่ทดสอบแล้ว |

## 14. นโยบายความสัมพันธ์กับผู้จัดหา

### 14.1 คำประกาศนโยบาย
ข้อกำหนดความปลอดภัยจะถูกกำหนดและรักษากับผู้จัดหาทั้งหมดที่เข้าถึง ประมวลผล หรือจัดเก็บข้อมูลของ Orbplus

### 14.2 การเลือกผู้จัดหา

#### 14.2.1 การประเมินความปลอดภัย

| ประเภทผู้จัดหา | การประเมินที่จำเป็น |
|--------------|------------------|
| ผู้ให้บริการ Cloud/โฮสติ้ง | แบบสอบถามความปลอดภัย, การรับรอง |
| ผู้ขายซอฟต์แวร์ | การทบทวนความปลอดภัย, ประวัติช่องโหว่ |
| พันธมิตรเอาท์ซอร์ส | ตรวจสอบประวัติ, ข้อตกลงด้านความปลอดภัย |
| บริการวิชาชีพ | NDA, การควบคุมการเข้าถึง |

#### 14.2.2 การรับรอง/มาตรฐานที่ต้องการ

| ประเภทบริการ | การรับรองที่ต้องการ |
|------------|------------------|
| โครงสร้างพื้นฐาน Cloud | ISO 27001, SOC 2 Type II |
| การประมวลผลการชำระเงิน | PCI DSS |
| การประมวลผลข้อมูล | ISO 27001, GDPR compliance |

### 14.3 ข้อกำหนดในสัญญา

ข้อตกลงกับผู้จัดหาทั้งหมดจะรวม:
- ข้อกำหนดความปลอดภัยของข้อมูล
- ภาระผูกพันการรักษาความลับ
- ข้อกำหนดการปกป้องข้อมูล
- สิทธิ์ในการตรวจสอบ
- ข้อกำหนดการแจ้งเหตุการณ์
- เงื่อนไขการสิ้นสุดและการคืนข้อมูล

### 14.4 การจัดการอย่างต่อเนื่อง

| กิจกรรม | ความถี่ | ความรับผิดชอบ |
|--------|--------|--------------|
| การทบทวนประสิทธิภาพ | รายไตรมาส | เจ้าของบริการ |

## 15. นโยบายการจัดการเหตุการณ์

### 15.1 คำประกาศนโยบาย

### 15.2 คำจำกัดความของเหตุการณ์

เหตุการณ์ด้านความปลอดภัยคือเหตุการณ์ใดๆ ที่:
- ละเมิดความลับ ความสมบูรณ์ หรือความพร้อมใช้งาน
- ละเมิดนโยบายความปลอดภัย
- บ่งชี้การเข้าถึงหรือการใช้โดยไม่ได้รับอนุญาต
- อาจส่งผลเสียต่อองค์กร

| **ปานกลาง (P3)** | ผลกระทบจำกัด | 4 ชั่วโมง |
